# Case Study: VS Code — PR #324192

## Repository
[microsoft/vscode](https://github.com/microsoft/vscode)

## PR
[#324192 — a11y: Update warning icon colors in 2026 Light theme](https://github.com/microsoft/vscode/pull/324192)

## Category
Contrast / theming

## What Happened
The "Autopilot (Preview)" label and other warning/info text in the Copilot permission-picker combo box failed the WCAG 1.4.3 minimum contrast ratio (4.5:1) against its background in the 2026 Light theme, making it hard to read for users with low vision or reduced contrast sensitivity. The issue was logged as a formal internal accessibility test finding (tagged `#A11yTCS`, `#WCAG1.4.3`, filed by a dedicated accessibility tester role) rather than a community bug report. The fix darkened the warning icon's foreground color to roughly 6:1 contrast and removed a CSS opacity reduction that had been lowering effective contrast further.

## Review Pattern Observed
This is the only case study in this set backed by a formal, named internal accessibility-testing pipeline (tagged issue, dedicated tester role, required verification-before-close) rather than an ad hoc community report — the PR review itself is almost a formality because the rigor happened upstream, at issue-filing time.

## Infrastructure Gap Illustrated
None, largely — this is closest to a model process, and the interesting finding is *why*: Microsoft has an internal accessibility QA function feeding structured, WCAG-tagged issues into the repo. That infrastructure isn't something a typical open-source project can replicate, but the shape of it — a required field for WCAG SC, and a rule that only a verified tester can close the bug — is exactly what a lightweight issue template could approximate for projects without a dedicated a11y team.
